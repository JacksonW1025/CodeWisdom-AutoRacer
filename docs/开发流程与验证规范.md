## 提交时机

满足以下条件后提交：

- 当前分支已经确认。
- 本次任务相关文件已经完整暂存。
- `msg`、node、launch、config、URDF、TF、odom、串口桥或 Nav2/SLAM 参数变化已经同步到根仓库对应文档。
- 影响阶段 1-4 验收的上位机改动已经按根 `docs/阶段路线图.md` 和 `docs/开发流程与验证规范.md` 的验收契约回填追踪表。
- 本仓库 README、launch/config、接口或流程文档变化后，已经检查根 `README.md`、`AGENTS.md` 和根 `docs/` 是否需要同步。
- 本仓库 README、`启动与运行规范.md` 或本文件已经同步本仓库流程变化。
- 当前机器构建验证已经通过，或文档-only 改动已经通过 `git diff --check`。
- 构建命令、结果和日志路径已经写入测试记录。

## 提交粒度

- message/interface 变化单独提交。
- 串口桥节点变化单独提交。
- launch/config 参数变化单独提交。
- URDF/TF/odom 几何语义变化单独提交。
- SLAM/Nav2 配置变化单独提交。
- 文档-only 修正单独提交。
- 跨下位机协议变化时，本仓库提交只承载上位机消息、桥接、launch/config 和本仓库文档；根仓库最后记录跨仓库文档和子模块指针。

## 本地验证

列出包名：

```bash
colcon list --base-paths src --names-only
```

全量构建：

```bash
colcon build --symlink-install
```

单包构建：

```bash
colcon build --symlink-install --packages-select <package_name>
```

文档和格式检查：

```bash
git diff --check
```

阶段 1 最小验收工具：

```bash
python3 tools/acceptance/stage1_acceptance_check.py --mode offline
python3 tools/acceptance/stage1_acceptance_check.py --mode live
python3 tools/acceptance/stage1_acceptance_check.py --mode live --require-odom
```

`--mode offline` 不打开串口、不发布运动命令；`--mode live` 只检查已运行 topic 和类型。`--require-odom` 用于阶段 1 odom 链路已启动的检查场景。

阶段 2 path fixture 检查：

```bash
python3 tools/acceptance/stage2_path_fixture_check.py
python3 tools/acceptance/stage2_tracker_fake_odom_check.py
```

当前阶段 2 已准备 fixture、Pure Pursuit tracker 和 fake-odom 离线检查。fake-odom 检查通过只代表软件契约满足；缺少
实车 `/odom`、`/ackermann_cmd`、`/path_tracking/diagnostics`、rosbag/log 和安全记录时，阶段 2 仍不得写成实车 PASS。

涉及 `/ackermann_cmd`、`/chassis_state`、`/wheel_odom`、`/odom`、TF、IMU、LiDAR、launch/config、Nav2/SLAM 参数或串口桥时，同步核对根仓库 `README.md`、`AGENTS.md`、`docs/阶段路线图.md`、`docs/接口与协议.md`、`docs/系统架构与数据流.md` 和 `docs/开发流程与验证规范.md`。

阶段 1-4 的测试记录必须使用根目录阶段验收契约中的 case 名、状态和证据要求。未实现 fixture、脚本、launch 或硬件条件时，记录为 `未实现`、`FAIL` 或 `BLOCKED`，不得静默跳过或写成 PASS。阶段 4 还必须记录 `/cmd_vel` 隔离证据和 `/twist_to_ackermann/diagnostics`。

每个阶段在标记最终 PASS 前，必须另开一个独立验证会话或独立 verifier 对话复核。复核会话不得发送运动命令、不得修改文件，只能读取记录、检查代码/配置、运行非运动自动检查和查看已有 rosbag/log。复核结论、证据和剩余风险必须写入阶段测试记录或最终回复；复核未通过时不得进入下一阶段。

## Ackermann bringup 与 odom 验收

当前默认 bringup 与 phase-1 验收入口分为两条路径：

- `turn_on_autoracer_robot.launch.py` 当前仍是 legacy/default 语义：`autoracer_robot` 订阅 `/cmd_vel`，发布 raw `/odom`；默认 `use_ekf:=true` 时 `autoracer_ekf.launch.py` 将 EKF 输出 remap 到 `/odom_combined`。
- legacy/default 路径可用于迁移、手动调试和旧链路兼容，但不能作为 phase-1 Ackermann 底盘或 canonical `/odom` 验收依据。
- 旧 `/cmd_vel` 不能作为正式 STM32 下行协议或阶段验收入口；当前兼容输入/桥接转换仅用于迁移、手动或 legacy 兼容。

`turn_on_autoracer_robot/launch/ackermann_chassis.launch.py` 当前默认值为：

```text
counts_per_meter:=0.0
use_ekf:=false
```

这个默认路径只用于串口、telemetry 和 `/chassis_state` bringup，不能作为阶段 1 可信 odom 验收路径。

阶段 1 odom 验收必须显式给出已标定的 `counts_per_meter>0`，并启动 `robot_localization` EKF：

```bash
ros2 launch turn_on_autoracer_robot ackermann_chassis.launch.py counts_per_meter:=<实测值> use_ekf:=true
```

验收记录必须说明 `counts_per_meter` 的来源、`/wheel_odom` 是否由 `hall_delta_count + steering_angle` 积分、`/odom` 是否由 `/wheel_odom + /imu/data` 的 EKF 输出。还必须记录 `base_footprint -> gyro_link` TF 存在、`/odom` 频率接近 `30 Hz`，以及静止时 `/odom` 不在无霍尔计数情况下持续漂移。launch 默认值改为验收路径时，同步更新本文件和根文档。

2D SLAM launch 默认 `odom_frame:=odom`，面向 phase-1 canonical `/odom`。legacy/default bringup 使用 `/odom_combined` 时，必须显式覆盖：

```bash
ros2 launch autoracer_slam_toolbox slam.launch.py odom_frame:=odom_combined
```

或先迁移到 `ackermann_chassis.launch.py counts_per_meter:=<实测值> use_ekf:=true` 的 Ackermann 验收路径。

跨层同步检查：

```bash
git diff --name-only -- README.md docs src
rg -n "<本次改动关键词>" ../README.md ../AGENTS.md ../docs README.md docs src || true
```

关键词按本次改动替换，例如 `/ackermann_cmd`、`/wheel_odom`、`/odom`、`TF`、`Nav2`、`feature/ackermann-chassis`。
